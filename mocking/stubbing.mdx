Stubbing tells MockK what a mocked method should do when it is called. You define stubs inside an `every { }` block before your code under test runs.

## The `every { }` block

```kotlin
val car = mockk<Car>()

every { car.drive(Direction.NORTH) } returns Outcome.OK
```

The lambda passed to `every` is a *recording scope*: MockK intercepts the call to detect which method and arguments you are configuring, then attaches the answer you specify.

For coroutines, use `coEvery` instead:

```kotlin
val car = mockk<Car>()

coEvery { car.drive(Direction.NORTH) } returns Outcome.OK
```

## Return values

### `returns`

Return a single fixed value:

```kotlin
every { car.drive(Direction.NORTH) } returns Outcome.OK
```

### `returnsMany`

Return values from a list in order. Each call advances to the next element; the last value repeats:

```kotlin
every { car.drive(any()) } returnsMany listOf(Outcome.OK, Outcome.RECORDED, Outcome.OK)
```

### `returnsArgument(n)`

Echo back the nth argument (zero-indexed):

```kotlin
every { mock.identity(any()) } returnsArgument 0
```

## Throwing exceptions

### `throws`

Throw a single exception:

```kotlin
every { car.drive(Direction.NORTH) } throws RuntimeException("engine failure")
```

### `throwsMany`

Throw exceptions in sequence, with the last repeating:

```kotlin
every { car.drive(any()) } throwsMany listOf(
    RuntimeException("first failure"),
    IllegalStateException("second failure")
)
```

## Lambda answers

Use `answers { }` when the return value depends on the arguments or requires custom logic. The block runs inside the **answer scope**, giving you access to call details.

```kotlin
val obj = mockk<MockedClass>()
val slot = slot<Int>()

every {
    obj.sum(any(), capture(slot))
} answers {
    1 + firstArg<Int>() + slot.captured
}

obj.sum(1, 2) // returns 4
obj.sum(1, 3) // returns 5
```

For coroutines:

```kotlin
coEvery { car.drive(any()) } coAnswers { computeOutcome() }
```

### Answer scope reference

Inside an `answers { }` block you have access to the following properties:

| Property | Description |
|---|---|
| `call` | The full call object (invocation + matcher). |
| `invocation` | Information about the actual function invoked. |
| `matcher` | The matcher used to match the invocation. |
| `self` | The object the call was made on. |
| `method` | The function that was invoked. |
| `args` | All invocation arguments as a list. |
| `nArgs` | Number of invocation arguments. |
| `arg(n)` | The nth argument (zero-indexed). |
| `firstArg()` | The first argument. |
| `secondArg()` | The second argument. |
| `thirdArg()` | The third argument. |
| `lastArg()` | The last argument. |
| `captured()` | The last element in a capture list. |
| `lambda<...>().invoke()` | Call a captured lambda. |
| `coroutine<...>().coInvoke()` | Call a captured coroutine. |
| `nothing` | A `null` sentinel for returning nothing. |
| `fieldValue` | The value in the property's backing field. |
| `fieldValueAny` | The backing field value typed as `Any?`. |
| `value` | The value being set (in property setter stubs). |
| `valueAny` | The setter value typed as `Any?`. |
| `callOriginal` | Calls the original (real) implementation. |

## Unit-returning functions

Three equivalent ways to stub a `Unit`-returning function so it does nothing:

```kotlin
justRun { obj.sum(any(), 3) }
every { obj.sum(any(), 3) } just Runs
every { obj.sum(any(), 3) } returns Unit
```

Full example:

```kotlin
class MockedClass {
    fun sum(a: Int, b: Int): Unit {
        println(a + b)
    }
}

val obj = mockk<MockedClass>()

justRun { obj.sum(any(), 3) }

obj.sum(1, 1)
obj.sum(1, 3)

verify {
    obj.sum(1, 1)
    obj.sum(1, 3)
}
```

### `just Awaits` for suspending functions

To simulate a suspending function that never returns:

```kotlin
runTest {
    val car = mockk<Car>()

    coJustAwait { car.drive(any()) } // car.drive(...) will never return

    val job = launch(UnconfinedTestDispatcher()) {
        car.drive(Direction.NORTH)
    }

    coVerify { car.drive(Direction.NORTH) }

    job.cancelAndJoin()
}
```

Equivalently: `coEvery { car.drive(any()) } just Awaits`

## Chained calls

Stub a chain of method calls in one `every` block:

```kotlin
val car = mockk<Car>()

every { car.door(DoorType.FRONT_LEFT).windowState() } returns WindowState.UP

car.door(DoorType.FRONT_LEFT) // returns a chained mock for Door
car.door(DoorType.FRONT_LEFT).windowState() // returns WindowState.UP

verify { car.door(DoorType.FRONT_LEFT).windowState() }

confirmVerified(car)
```

<Note>
If the intermediate return type is generic, type information may be erased. Use `hint` to supply it explicitly:

```kotlin
every { obj.op2(1, 2).hint(Int::class).op1(3, 4) } returns 5
```
</Note>

## Additional answers

Chain multiple answers together. Each subsequent call advances to the next; the last answer repeats.

| Additional answer | Description |
|---|---|
| `andThen value` | Return the specified value on the next call. |
| `andThenMany list` | Return values from the list in order. |
| `andThenThrows ex` | Throw an exception on the next call. |
| `andThenThrowsMany list` | Throw exceptions from the list in order. |
| `andThen { code }` | Execute a code block on the next call. |
| `coAndThen { code }` | Execute a coroutine code block on the next call. |
| `andThenAnswer answerObj` | Use an `Answer` object on the next call. |
| `andThen { nothing }` | Return `null` on the next call. |
| `andThenJust Runs` | Return `Unit` on the next call. |
| `andThenJust Awaits` | Never return on the next call (since v1.13.3). |

Example:

```kotlin
every { car.drive(any()) } returns Outcome.OK andThen Outcome.RECORDED andThenThrows RuntimeException("done")
```

## Answers table

| Answer | Description |
|---|---|
| `returns value` | Return a fixed value. |
| `returnsMany list` | Return values in sequence; last repeats. |
| `returnsArgument(n)` | Return the nth argument. |
| `throws ex` | Throw an exception. |
| `throwsMany list` | Throw exceptions in sequence; last repeats. |
| `answers { code }` | Execute a code block in answer scope. |
| `coAnswers { code }` | Execute a coroutine code block in answer scope. |
| `answers answerObj` | Use a custom `Answer` object. |
| `answers { nothing }` | Return `null`. |
| `just Runs` | Return `Unit` (for `Unit`-returning functions). |
| `just Awaits` | Never return (for suspending functions, since v1.13.3). |
| `propertyType Class` | Specify type of the backing field accessor. |
| `nullablePropertyType Class` | Specify nullable type of the backing field accessor. |

## Property backing fields

You can read and write a property's backing field inside an answer block using `fieldValue` and `value`.

<Note>
This is only for public properties. Private property backing fields cannot be mocked directly. Use Java reflection or `@VisibleForTesting` to make them accessible.
</Note>

```kotlin
val mock = spyk(MockCls(), recordPrivateCalls = true)

every { mock.property } answers { fieldValue + 6 }
every { mock.property = any() } propertyType Int::class answers { fieldValue += value }
every { mock getProperty "property" } propertyType Int::class answers { fieldValue + 6 }
every { mock setProperty "property" value any<Int>() } propertyType Int::class answers { fieldValue += value }
every {
    mock.property = any()
} propertyType Int::class answers {
    fieldValue = value + 1
} andThen {
    fieldValue = value - 1
}
```

Use `nullablePropertyType` for nullable property types.
